# SalesOn 재분석 기반 재베이스라인 로드맵

## 목적

- 현재 `vibe-shop` 저장소의 실제 구현 범위를 다시 고정한다.
- `C:\Users\Administrator\Desktop\세일즈온_솔루션` 아래 원본 레거시 자료를 직접 읽어 1차/2차 범위를 다시 잡는다.
- 앞으로의 개발을 `신규 구축` 원칙에 맞는 현실적인 단계로 나눈다.

## 이번 재분석 기준

### 현재 저장소에서 확인한 사실

- Git 히스토리는 2026-03-13 단일 커밋 1개다.
- 사용자 앱은 `apps/storefront`, API는 `apps/api`, 관리자 앱은 `apps/admin` 구조로 분리되어 있다.
- 현재 사용자 라우트는 홈, 카테고리 상세, 상품 상세, 장바구니, 체크아웃, 주문완료 중심이다.
- 현재 API 엔드포인트는 `home`, `categories`, `products`, `product detail`, `order preview`, `order create`, `order get` 정도다.
- 장바구니는 서버 저장이 아니라 `localStorage` 기반이다.
- 주문 생성은 금액 계산과 주문/주문라인 저장까지이며, 재고 차감, 상태 머신, 결제 승인, 중복 제출 방지가 없다.
- 백엔드 테스트는 `contextLoads()` 1건뿐이고, 테스트 설정은 `H2 + create-drop + Flyway off`다.
- E2E는 `tests/e2e/mvp-smoke.js` 1개로 홈에서 주문완료까지 smoke만 검증한다.

### 레거시 원본에서 다시 확인한 사실

- 분석 기준 폴더는 아래 세 개다.
  - `세일즈온_산출물`
  - `세일즈온_API 문서`
  - `saleson-solution-saleson-3.13.2`
- 메뉴 구조 엑셀 원본은 깨지지 않은 상태로 읽힌다.
- 사용자 메뉴 시트에서 `Program` 기준으로 확인 가능한 사용자 프로그램은 39개다.
- 관리자/판매자 시트도 살아 있으며, 2차 범위를 별도 트랙으로 나누기에 충분한 수준이다.
- API 문서 원본 `api_data.js`에서 그룹별 엔드포인트 수를 다시 확인할 수 있다.

## 핵심 판단

### 1. 현재 프로젝트는 맞는 방향의 초기 MVP다

- 구조는 신규 구축 원칙과 맞다.
- 프런트/백 분리, API 중심, PostgreSQL/Flyway 기준도 맞다.
- typed client 분리도 방향이 좋다.

### 2. 하지만 아직 운영 가능한 커머스 코어는 아니다

- 주문이 저장만 되고 실제 주문 도메인으로 굴러가지는 않는다.
- 인증이 없어서 회원/비회원 퍼널과 마이페이지가 이어지지 않는다.
- 검색/전시/리뷰/문의/위시리스트가 비어 있어 쇼핑몰 체감이 약하다.
- 테스트와 실행 재현성이 부족하다.

### 3. 다음 단계는 화면 확장보다 도메인 하드닝이 먼저다

- 현재 checkout을 더 화려하게 만드는 것보다
  - 인증
  - 서버 장바구니
  - 주문 상태
  - 재고 처리
  - 결제 어댑터
  - 테스트 체계
  를 먼저 붙여야 한다.

## 재베이스라인 로드맵

### Phase 0. 개발 기반 안정화

목표: 팀이 같은 방식으로 실행, 검증, 배포 준비를 할 수 있게 만든다.

- 루트 README 추가
- 실행 가이드 정리
- Windows 전용 루트 스크립트 제거 또는 교차 플랫폼화
- `compose.yaml` 확장
  - Postgres 외에 API/storefront 실행 전략 정리
- 환경변수 정책 정리
  - `API_BASE_URL`
  - `NEXT_PUBLIC_API_BASE_URL`
  - DB 계정
- OpenAPI 또는 계약 문서 초안 작성
- ERD 초안 작성
- CI 초안
  - storefront lint/build
  - api test
  - Playwright smoke

완료 기준:

- 새 팀원이 문서만 보고 로컬 실행 가능
- 최소 CI가 실패/성공을 구분 가능
- API base URL 전략이 브라우저/서버 양쪽에서 명확함

### Phase 1. 구매 퍼널 MVP를 운영형 구조로 보강

목표: 현재 데모 퍼널을 실제 커머스 코어에 가까운 구조로 바꾼다.

- 인증/세션
  - 회원 로그인
  - 회원가입
  - 비회원 주문조회
  - 게스트/회원 주문 분기
- 서버 장바구니
  - localStorage 단독 구조 탈피
  - 로그인 전후 장바구니 머지 정책
- 주문 도메인 하드닝
  - 주문 상태 머신
  - idempotency
  - 재고 차감 또는 예약
  - 실패/취소 흐름
  - 결제 어댑터 인터페이스
- 주문 조회 확장
  - 목록
  - 상세
  - 기본 클레임 진입점

완료 기준:

- 회원/비회원 모두 구매 플로우 가능
- 중복 제출과 재고 불일치의 기본 방어 가능
- 주문 완료가 단순 레코드 생성이 아닌 상태 있는 주문으로 관리됨

### Phase 1A. 탐색과 상품 경험 확장

목표: 사용자가 실제 쇼핑몰처럼 탐색할 수 있게 만든다.

- 검색 결과 페이지
- 정렬/필터
- 전시 블록
  - 베스트
  - 신상품
  - 기획전
- 상품 이미지/미디어 구조
- 상품 상세 확장
  - 리뷰
  - 상품문의
  - 관련상품
  - 위시리스트

완료 기준:

- 홈/카테고리/검색/상세가 서로 연결됨
- 상품 탐색과 구매 진입이 자연스럽게 이어짐

### Phase 1B. 마이페이지와 콘텐츠

목표: 1차 사용자 범위를 실질적으로 닫는다.

- 마이페이지 메인
- 주문 목록/상세
- 배송지 관리
- 쿠폰/포인트
- 위시리스트
- 리뷰/문의
- 회사소개, 공지, FAQ, 정책, 입점문의

완료 기준:

- 회원이 로그인 후 주문 이후 흐름까지 수행 가능
- 문서형/고객센터형 페이지가 기본 제공됨

### Phase 2. 관리자 앱

목표: storefront와 API가 안정화된 뒤 운영 기능을 별도 앱으로 구축한다.

- 상품 관리
- 전시 관리
- 주문/클레임 관리
- 회원/권한 관리
- 판매자/정산

원칙:

- 사용자 앱과 관리자 앱은 별도 프런트엔드로 유지
- storefront P0 안정화 전에는 관리자 본격 착수 금지

## 디자인/프런트 작업 운영 메모

- UI 작업은 `frontend-design` 기준으로 진행한다.
- React 구현은 `react-best-practices` 기준으로 검토한다.
- 두 스킬은 설치 완료 상태이며, 이후 턴에서 활용하려면 Codex 재시작이 필요하다.
- 디자인은 지금의 MVP 스타일을 유지 복제하기보다, 커머스 전시/탐색/구매 흐름에 맞는 새 UI 시스템으로 확장하는 방향이 적절하다.

## 최종 정리

- 현재 상태는 `구매 퍼널 데모가 되는 초기 MVP`다.
- 레거시 원본 재분석 결과, 1차 사용자 범위는 이전 판단처럼 넓고 현재 구현은 그 일부에 해당한다.
- 다음 개발은 화면 수를 급히 늘리기보다 `기반 안정화 -> 인증/장바구니/주문 하드닝 -> 검색/전시 -> 마이페이지` 순서가 맞다.
- 관리자와 판매자 기능은 storefront P0 안정화 이후 별도 트랙으로 가는 것이 안전하다.
