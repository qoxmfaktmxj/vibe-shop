# Phase 2 운영/참여 기능 확장 계획서
작성일: 2026-03-17
대상 저장소: `vibe-shop`

## 목적

- 현재 완료된 사용자/관리자 MVP 이후 남은 핵심 운영 기능과 고객 참여 기능의 구현 순서를 고정한다.
- `배너/전시 CRUD`, `카테고리 관리`, `회원 관리`, `통계`와 `리뷰`, `찜`, `쿠폰`, `기획전`, `상품 문의`, `CS`를 한 번에 섞지 않고 단계별로 나눠서 진행한다.
- 이 문서는 `구현 시작 전 PM 승인용 계획서`다.

## 현재 코드 기준 요약

### 이미 있는 것

- 사용자 인증/세션
- 회원/비회원 주문 분기
- 주문 상태, 결제 어댑터 뼈대, 재고 처리
- 상품/카테고리 조회, 검색, 홈 전시 기본 구조
- 마이페이지 기본 구조
- 관리자 로그인, 상품 수정, 주문 상태 변경, 메인 카피 수정

### 현재 경계

- 전시 데이터는 `admin_display_settings` 단일 행으로 `hero_title`, `hero_subtitle`만 관리한다.
- 카테고리는 `slug`, `name`, `description`, `accent_color`만 있고 노출 여부, 정렬 순서, 이미지 같은 운영 필드는 없다.
- 회원은 `users` 테이블 기준으로 `name`, `email`, `password_hash`, `provider`, `role`, `created_at` 정도만 있다.
- 통계는 관리자 대시보드에서 단순 count 집계만 제공한다.
- 리뷰, 찜, 쿠폰, 기획전, 상품 문의, CS 관련 API/테이블/프런트 화면은 아직 없다.

## 왜 이 순서로 가야 하는가

아래 기능들은 독립처럼 보이지만 실제로는 의존성이 있다.

1. 전시와 카테고리를 먼저 잡아야 관리자 앱 구조와 홈/카테고리 편집 흐름이 안정된다.
2. 회원 관리와 통계를 먼저 넣어야 이후 리뷰/찜/문의/CS를 운영할 대상 식별이 가능해진다.
3. 리뷰와 찜은 회원/상품 관계가 선행돼야 한다.
4. 쿠폰과 기획전은 주문 총액, 할인 계산, 홈/검색/상품 진열 규칙에 영향을 준다.
5. 상품 문의와 CS는 회원, 주문, 상품, 관리자 권한 모델이 있어야 완결된다.

## 개발 묶음 제안

### A. 운영 기반 확장

범위:

- 배너/전시 CRUD
- 카테고리 관리
- 회원 관리
- 통계

목표:

- 관리자 앱을 `운영 도구` 수준으로 올린다.
- storefront 메인/카테고리 화면을 관리자 데이터로 제어할 수 있게 만든다.

#### A1. 배너/전시 CRUD

현재 상태:

- 메인 카피 1세트만 수정 가능
- 섹션 개념 없음
- 배너 이미지, 링크, 노출 기간, 정렬, 공개 여부 없음

추가할 것:

- `display_sections`
- `display_items`
- 전시 타입 enum: `HERO`, `FEATURED_CATEGORY`, `CURATED_PICK`, `NEW_ARRIVALS`, `BEST_SELLERS`, `PROMOTION`
- 배너 필드: 제목, 설명, 이미지, 링크, 버튼 문구, 정렬 순서, 노출 여부, 노출 기간
- 관리자 전시 목록/상세/생성/수정/삭제 API
- storefront 홈 응답을 단일 hero 문구 기반에서 섹션 기반 응답으로 확장

관리자 UI:

- 전시 목록
- 전시 섹션별 아이템 정렬
- 배너 생성/수정 폼
- 공개/비공개 토글

QA 포인트:

- 공개 중인 전시만 홈에 노출되는지
- 정렬/노출 기간이 반영되는지
- 잘못된 링크, 비어 있는 이미지 등 입력 검증이 되는지

#### A2. 카테고리 관리

현재 상태:

- 읽기 전용
- 운영용 속성 부족

추가할 것:

- `categories` 확장 필드:
  - `display_order`
  - `is_visible`
  - `cover_image_url`
  - `cover_image_alt`
  - `hero_title`
  - `hero_subtitle`
- 카테고리 CRUD API
- 카테고리 정렬 변경 API
- 카테고리 비노출 처리 시 storefront 목록 반영

관리자 UI:

- 카테고리 목록
- 생성/수정/삭제
- 노출 여부, 정렬 순서 변경

QA 포인트:

- 비노출 카테고리가 사용자 화면에서 사라지는지
- 정렬 순서가 헤더/홈/목록에 일관되게 반영되는지

#### A3. 회원 관리

현재 상태:

- 세션 및 회원가입은 있으나 운영 화면 없음
- 회원 상태, 마지막 로그인, 마케팅 수신 동의 같은 운영 필드 없음

추가할 것:

- `users` 확장 필드:
  - `status`
  - `phone`
  - `last_login_at`
  - `marketing_opt_in`
- 관리자 회원 목록/상세 API
- 필터: 가입일, provider, role, status
- 회원 상태 변경: `ACTIVE`, `DORMANT`, `BLOCKED`

관리자 UI:

- 회원 목록
- 회원 상세
- 상태 변경
- 주문/배송지/찜/리뷰/문의 요약 탭

QA 포인트:

- 차단 회원의 로그인/주문/찜 등 제한 정책이 일관적인지
- 소셜 회원과 일반 회원이 동일한 목록/필터에서 보이는지

#### A4. 통계

현재 상태:

- 단순 count 위주 대시보드

추가할 것:

- 일 단위 집계 API
- 최근 7일/30일:
  - 주문 수
  - 결제 완료 금액
  - 신규 회원 수
  - 취소/환불 건수
  - 카테고리별 판매
  - 상위 상품
- 초기에는 이벤트 트래킹 없이 주문/회원 데이터를 기준으로 파생 집계

관리자 UI:

- KPI 카드
- 일별 추이 차트
- 카테고리/상품 랭킹

QA 포인트:

- 주문 취소/환불이 매출 집계에 어떻게 반영되는지
- 날짜 범위/타임존 계산이 맞는지

## 왜 이 순서가 현실적인가

- 전시/카테고리는 현재 storefront와 관리자 앱 모두에 바로 효과가 있다.
- 회원 관리와 통계는 운영자가 시스템을 보는 기본 도구라 먼저 있어야 한다.
- 리뷰/찜은 사용자 경험 개선이 크지만 주문 계산을 깨지 않는다.
- 쿠폰은 주문 계산과 결제 미리보기를 건드리므로 그다음에 두는 게 안전하다.
- 문의/CS는 도메인 수가 많아 마지막에 넣는 편이 리스크가 낮다.

## 공통 선행 작업

- 공통 테이블 레이아웃과 검색/필터 UI 패턴 정리

  - 2차: 외부 스토리지(S3 등)
- 관리자 권한 체크 일원화
  - `OWNER`, `MD`, `CS`, `OPS`

- 문의 등록 -> 관리자 답변 -> 사용자 확인

## 리스크

- 쿠폰은 주문/결제 계산과 강하게 연결돼 있어서 별도 승인 없이 다른 묶음과 섞으면 위험하다.
- 전시 CRUD는 이미지/링크 정책을 먼저 정하지 않으면 나중에 데이터 구조를 다시 바꾸게 된다.
- 회원 관리는 계정 차단 정책과 주문/찜/리뷰 접근 정책을 같이 정의해야 한다.
- 통계는 추후 이벤트 추적을 붙일 수 있으므로 1차는 `주문/회원 기반 집계`로 제한하는 것이 안전하다.
- CS는 범위가 쉽게 커지므로 1차는 티켓과 답변 로그까지만 두는 것이 좋다.

## PM 승인 요청안

바로 시작할 첫 범위는 아래가 적절하다.

1. 승인 묶음 1부터 착수
2. 구현 순서는 `배너/전시 CRUD -> 카테고리 관리`
3. 완료 후 브라우저 QA까지 확인하고 다음 묶음 승인 여부를 결정

이유:

- 현재 관리자 앱에 가장 자연스럽게 이어붙일 수 있다.
- storefront 가시 변화가 크고, 이후 기획전/배너/홈 운영의 기반이 된다.
- 주문/결제 같은 고위험 영역을 건드리지 않고도 운영 수준을 올릴 수 있다.
