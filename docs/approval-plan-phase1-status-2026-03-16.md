# Phase 1 상태 점검 및 승인용 계획서

작성일: 2026-03-16  
대상 저장소: `vibe-shop`

## 목적

- 현재 코드 기준으로 아래 6개 범위의 구현 여부를 판정한다.
- 미완료 범위는 승인 후 바로 착수할 수 있도록 실행 계획으로 정리한다.
- 이 문서는 `구현 시작 전 PM 승인용` 문서다.

대상 범위:

1. 인증과 세션
2. 회원/비회원 주문 분기
3. 주문 상태 확장, 재고, 결제 어댑터
4. 검색과 전시 확장
5. 마이페이지
6. 관리자 앱

## 권장 콤보 데이터 한글 기준

아래 코드는 화면 셀렉트, 상태 뱃지, 관리자 콤보박스, API 응답 공통 라벨 기준으로 사용한다.

### 주문 상태

| 코드 | 한국어 라벨 | 설명 |
| --- | --- | --- |
| `RECEIVED` | 주문 접수 | 현재 구현 상태값. 향후 `PENDING_PAYMENT`로 대체 가능. |
| `PENDING_PAYMENT` | 결제 대기 | 주문은 생성됐지만 결제 전인 상태 |
| `PAID` | 결제 완료 | 결제 승인 완료 |
| `PREPARING` | 상품 준비중 | 출고 전 피킹/패킹 단계 |
| `SHIPPED` | 배송중 | 송장 발급 및 출고 완료 |
| `DELIVERED` | 배송 완료 | 고객 수령 완료 |
| `CANCELLED` | 주문 취소 | 주문 취소 완료 |
| `REFUND_REQUESTED` | 환불 요청 | 환불 접수 상태 |
| `REFUNDED` | 환불 완료 | 환불 처리 완료 |

권장 방향:

- 지금 `RECEIVED`는 한국어로 `주문 접수`로 번역한다.
- 실제 결제 도입 시에는 신규 주문 기본값을 `PENDING_PAYMENT`로 바꾸고 `RECEIVED`는 제거하거나 마이그레이션한다.

## 상세 판정

### 1. 인증과 세션

현재 있는 것:

- 장바구니 세션 쿠키 `vibe_shop_cart`
- HTTP-only 쿠키 기반 게스트 장바구니 유지

현재 없는 것:

- 회원 테이블
- 로그인 API
- 비밀번호 저장/검증
- 인증 세션 또는 토큰
- 권한 체크
- 회원 기준 서버 세션 머지

판정:

- `부분 구현`

### 2. 회원/비회원 주문 분기

현재 있는 것:

- 비회원 주문 생성
- 비회원 주문번호 + 휴대폰 기반 조회
- 휴대폰 기준 주문 이력 조회

현재 없는 것:

- 회원 주문 소유자 연결
- 로그인 사용자 주문 조회
- 회원/비회원 체크아웃 진입 분기
- 로그인 후 장바구니/주문 이관

판정:

- `미구현`

### 3. 주문 상태 확장, 재고, 결제 어댑터

주문 상태:

- 현재 상태값은 `RECEIVED`, `CANCELLED`만 존재
- 배송 준비, 결제 대기, 결제 완료, 배송중, 배송 완료, 환불/반품 등 없음

재고:

- 주문 생성 시 `decreaseStock`
- 주문 취소 시 `increaseStock`
- 최소 수준 재고 차감/복원은 있음

결제:

- 결제 인터페이스 없음
- PG 어댑터 없음
- 결제 승인/실패/웹훅 처리 없음

판정:

- 주문 상태: `부분 구현`
- 재고: `구현`
- 결제 어댑터: `미구현`

### 4. 검색과 전시 확장

검색:

- 키워드 검색 있음
- 카테고리 + 키워드 조합 있음
- 정렬은 `recommended`, `price-asc`, `price-desc` 정도만 있음

전시:

- 홈 featured 카테고리/상품 노출 있음
- 카테고리 목록/상품 목록/상세 있음
- 배너, 기획전, 큐레이션 슬롯, 전시 관리 구조 없음

판정:

- 검색: `기본 구현`
- 전시 확장: `부분 구현`

### 5. 마이페이지

현재 없는 것:

- 마이페이지 메인
- 회원 주문 목록/상세
- 배송지 관리
- 프로필 수정
- 찜/최근본/쿠폰/포인트

판정:

- `미구현`

### 6. 관리자 앱

현재 상태:

- `apps/admin`에 README만 있음
- 앱 라우트, 레이아웃, 상태 관리, API 클라이언트 모두 없음

## 승인 후 구현 우선순위

권장 순서:

1. 인증과 세션
2. 회원/비회원 주문 분기
3. 주문 상태 확장 + 결제 어댑터 경계
4. 검색과 전시 확장
5. 마이페이지
6. 관리자 앱

이 순서가 맞는 이유:

- 마이페이지와 관리자 앱은 회원 인증과 주문 모델이 먼저 잡혀야 한다.
- 결제 어댑터 없이 주문 상태를 확장해도 실제 결제 이후 흐름이 닫히지 않는다.
- 검색/전시는 인증보다 독립적이지만, 구매 코어보다 우선순위는 낮다.
- 관리자 앱은 storefront와 API 도메인이 안정화된 뒤 시작하는 편이 안전하다.

## 승인 요청 범위

이번 승인에서는 아래 3개 묶음을 먼저 착수 범위로 제안한다.

### 묶음 A. 인증과 세션

범위:

- 회원 엔티티/리포지토리/마이그레이션
- 회원가입
- 로그인/로그아웃
- 세션 쿠키
- 현재 게스트 장바구니와 로그인 회원 장바구니 머지 정책

완료 기준:

- 비회원도 기존 장바구니 사용 가능
- 로그인 후 회원 기준으로 장바구니와 주문 접근 가능
- 서버와 storefront 모두 인증 상태를 판단할 수 있음

QA 포인트:

- 회원가입 후 로그인 가능
- 로그아웃 시 보호 페이지 접근 차단
- 로그인 전후 장바구니 데이터 손실 없음

### 묶음 B. 회원/비회원 주문 분기 + 주문 상태/결제 경계

범위:

- 체크아웃 시 회원/비회원 분기
- 회원 주문 소유권 연결
- 주문 상태 확장
- 결제 어댑터 인터페이스 추가
- 더미 결제 어댑터로 승인/실패 흐름 검증
- 재고 차감/복원 규칙 점검

권장 주문 상태 초안:

- `PENDING_PAYMENT`
- `PAID`
- `PREPARING`
- `SHIPPED`
- `DELIVERED`
- `CANCELLED`
- `REFUND_REQUESTED`
- `REFUNDED`

완료 기준:

- 회원은 본인 주문만 조회 가능
- 비회원은 주문번호 + 휴대폰으로만 조회 가능
- 결제 성공/실패에 따라 주문 상태가 일관되게 바뀜
- 중복 제출과 재고 불일치에 대한 기본 방어가 있음

QA 포인트:

- 회원 주문, 비회원 주문 각각 생성 가능
- 실패 결제 시 재고와 주문 상태가 정상 복구
- 중복 결제 요청 방지 확인

### 묶음 C. 검색과 전시 확장

범위:

- 검색 필터 확장
- 홈 전시 섹션 확장
- 카테고리별 추천/신상품/베스트 분리
- 기획전 또는 editorial 컬렉션 구조 초안

완료 기준:

- 사용자가 검색에서 상세 진입까지 자연스럽게 이동 가능
- 홈과 카테고리 화면에 전시 블록이 명확히 구분됨

QA 포인트:

- 정렬/검색 결과 일관성
- 전시 섹션과 실제 상품 연결 정확성

## 리스크

- 현재 코드는 주문과 회원 도메인이 분리되어 있어서 인증을 넣는 순간 계약 변경 폭이 커진다.
- 결제 어댑터는 인터페이스부터 두고 더미 구현으로 검증한 뒤 실제 PG를 붙여야 한다.
- 관리자 앱은 2차 범위이므로 storefront 핵심 도메인이 안정화되기 전 착수하면 중복 변경이 많아진다.
